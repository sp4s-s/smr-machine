## Benchmark Parameters Explained

### Capacity
**What it is:** The size (number of elements) of the single-producer/single-consumer (SPSC) ring buffer.

- Must be a **power of two** (e.g., 512, 1024, 2048)
- Default: 1024
- Each element is a `LossEvent` (32 bytes)
- The ring buffer uses **cache-line isolation** (64-byte alignment) to prevent false sharing between producer and consumer

**Example:** `--capacity 1024` means the queue can hold 1024 events before the producer blocks or the oldest events are overwritten (depending on configuration).

### Lanes
**What it is:** The number of independent SPSC ring buffer pairs running in parallel.

- Each lane has its own producer and consumer thread
- Default: 1 (single lane)
- Multiple lanes allow testing throughput under higher concurrency
- Each lane operates independently but shares CPU resources

**Example:** `--lanes 4` means 4 separate producer-consumer pairs running concurrently, effectively 4x the throughput potential (depending on CPU cores).

### Stress Threads
**What it is:** The number of background threads that add memory and scheduler pressure without directly participating in the queue operations.

- Default: 0
- These threads allocate memory and perform operations to simulate realistic system load
- They help test how the queue performs under thermal pressure and system noise

**Example:** `--stress-threads 2` adds 2 background threads that compete for CPU resources, simulating real-world conditions.

### Stress Bytes
**What it is:** The amount of memory (in bytes) allocated by each stress thread.

- Default: 8 MB (8 * 1024 * 1024 = 8,388,608 bytes)
- Each stress thread allocates this amount of memory to create memory pressure
- Higher values = more memory pressure = more realistic latency distributions

**Example:** `--stress-bytes 33554432` (32 MB) per stress thread creates significant memory pressure.

### Report Interval (ms)
**What it is:** The time interval (in milliseconds) between each timeline sample output.

- Default: 250 ms
- Controls how frequently throughput metrics are recorded
- Lower values = more granular data but larger output files

**Example:** `--report-interval-ms 1000` records metrics every 1 second.

### Duration (sec)
**What it is:** How long the benchmark runs.

- Default: depends on messages or explicit duration
- Can be set via `--duration-sec` or `--messages`

**Example:** `--duration-sec 60` runs the benchmark for 60 seconds.

### Latency Bucket (ns)
**What it is:** The size of each histogram bucket for latency distribution.

- Default: 100 ns
- Controls granularity of latency histogram

**Example:** `--latency-bucket-ns 50` gives finer granularity in the 0-50ns, 50-100ns, etc. ranges.

### Latency Max (ns)
**What it is:** The maximum latency tracked in the histogram.

- Default: 100,000,000 ns (100 ms)
- Latencies beyond this are counted in the final bucket

**Example:** `--latency-max-ns 50000000` tracks latencies up to 50ms.

## Timeline CSV Format

The timeline CSV file contains interval throughput samples recorded during the benchmark run.

### Columns:
| Column | Description |
|--------|-------------|
| `elapsed_ms` | Time elapsed since start (in milliseconds) |
| `produced` | Total number of messages produced so far |
| `consumed` | Total number of messages consumed so far |
| `interval_mps` | Throughput in this interval (millions of messages per second) |

### Example Output:
```csv
elapsed_ms,produced,consumed,interval_mps
205.019,1945420,1943295,9.727
401.297,3869302,3866321,9.619
604.507,5846689,5845572,9.887
808.129,7708043,7706087,9.307
1004.986,9539584,9539584,9.158
```

### Use Cases:
- Plotting throughput drift over time
- Detecting thermal throttling
- Identifying burst patterns
- Analyzing backlog growth

## Examples Directory

The `examples/` directory contains sample event scripts for the deterministic replay system.

### Files:

#### 1. `loss_scenario.pr42`
A predefined scenario demonstrating an expensive-loss incident with orders and a synthetic failure.

```pr42
# Deterministic replay scenario for an expensive-loss incident.
SUBMIT 1000000 ord-100 alpha BUY 8 100
FILL 1000010 ord-100 3 100
SUBMIT 1000020 ord-101 beta SELL 5 101
FILL 1000030 ord-101 5 101
SUBMIT 1000040 ord-102 alpha SELL 6 99
FILL 1000050 ord-102 6 99
FAIL 1000060 venue-disconnect-loss-check
```

#### 2. `generated_from_cpp.pr42`
A scenario generated by the C++ `generate_scenario` tool, demonstrating cross-language integration.

```pr42
# generated by cpp/src/generate_scenario.cpp
SUBMIT 2000000 ord-201 alpha BUY 12 100
FILL 2000010 ord-201 7 100
SUBMIT 2000020 ord-202 alpha SELL 10 103
MODIFY 2000025 ord-202 8 104
FILL 2000030 ord-202 8 104
SUBMIT 2000040 ord-203 beta SELL 5 99
CANCEL 2000050 ord-203
FAIL 2000060 synthetic-loss-boundary-breach
```

### Event Types:
| Event | Description |
|-------|-------------|
| `SUBMIT <ts> <id> <trader> <BUY|SELL> <qty> <px>` | Submit a new order |
| `MODIFY <ts> <id> <qty> <px>` | Replace the remaining open quantity and working price |
| `FILL <ts> <id> <qty> <px>` | Order filled (partial or full) |
| `CANCEL <ts> <id>` | Cancel an order |
| `FAIL <ts> <reason...>` | Inject a failure event for rollback testing |

## Generating Trace Files

### A. For Perfetto & Jane Street Magic Trace (trace_event.json)

#### Basic Generation:
```bash
# Ensure the C++ benchmark is built
make cpp

# Generate trace file with default settings
./build/spsc_bench \
  --duration-sec 30 \
  --capacity 1024 \
  --lanes 4 \
  --stress-threads 2 \
  --stress-bytes 33554432 \
  --report-interval-ms 250 \
  --trace-event-json trace_event.json
```

#### Using Built-in Scripts:
```bash
# Generate performance report (includes trace_event.json)
make perf-report

# Output location: build/perf_report/trace_event.json
```

#### Custom Output Directory:
```bash
mkdir -p traces
./build/spsc_bench \
  --duration-sec 60 \
  --capacity 1024 \
  --lanes 4 \
  --stress-threads 2 \
  --stress-bytes 33554432 \
  --report-interval-ms 1000 \
  --trace-event-json traces/my_trace.json
```

#### Mixed Command Stress With Failure Threshold:
```bash
mkdir -p build/command_stress_artifacts
./build/command_stress \
  --duration-sec 60 \
  --capacity 1024 \
  --lanes 4 \
  --stress-threads 2 \
  --stress-bytes 33554432 \
  --failure-limit 5 \
  --submit-weight 40 \
  --modify-weight 15 \
  --cancel-weight 15 \
  --fill-weight 25 \
  --fail-weight 5 \
  --invalid-bps 250 \
  --report-interval-ms 250 \
  --summary-json build/command_stress_artifacts/summary.json \
  --timeline-csv build/command_stress_artifacts/timeline.csv \
  --trace-event-json build/command_stress_artifacts/trace_event.json \
  --raw-events-jsonl build/command_stress_artifacts/raw_events.jsonl \
  --replay-dir build/command_stress_artifacts/replays
```

Artifacts:

- `summary.json`: mean, stddev, coefficient of variation, jitter, p50/p75/p90/p95/p99/p999, backlog, failures
- `timeline.csv`: throughput, moving average, backlog, success/failure counters
- `trace_event.json`: upload to Perfetto or Magic Trace
- `raw_events.jsonl`: live event stream for a separate order-terminal process
- `replays/failure_*.pr42`: replayable windows around the first failures

### B. For macOS Instruments (.trace files)

#### Using Built-in Script:
```bash
# Run profile script which generates .trace files
./scripts/profile_bench.sh

# Output will be in: build/spsc_bench.trace
```

#### Manual xctrace Command:
```bash
# Ensure build exists
make cpp

# Generate trace with xctrace directly
xctrace record \
  --template "Time Profiler" \
  --output build/manual_trace.trace \
  --launch -- ./build/spsc_bench \
    --duration-sec 30 \
    --capacity 1024 \
    --lanes 4 \
    --stress-threads 2 \
    --stress-bytes 33554432 \
    --report-interval-ms 250 \
    --timeline-csv build/timeline.csv
```

## Viewing Traces

### A. Viewing in Perfetto

#### Online Viewer (Recommended):
1. Open your browser to **https://ui.perfetto.dev/**
2. Click **"Open trace file"**
3. Select your `trace_event.json` file
4. The visualization will load automatically

#### What you'll see:
- Timeline of producer/consumer activity
- Per-lane event traces
- Duration and timing information
- Latency distributions

---

### B. Viewing in Jane Street Magic Trace

#### Web Interface:
1. Go to **https://magic-trace.org/**
2. Click **"Upload Trace"** or drag and drop your `trace_event.json` file
3. Wait for processing

**Note:** Compatibility is NOT guaranteed from macOS-generated artifacts. The project documentation states: *"compatibility is not guaranteed from macOS-generated artifacts, but this JSON is the portable trace to try first."*

---

### C. Viewing in macOS Instruments

#### Using GUI:
```bash
# Open the trace file in Instruments
open build/spsc_bench.trace

# Or specify Instruments directly
instruments -open build/spsc_bench.trace
```

#### Using Command Line:
```bash
# List available templates
xctrace list --template "Time Profiler"

# Examine trace data
xctrace examine build/spsc_bench.trace

# Export trace data
xctrace export build/spsc_bench.trace --output exported_trace.data
```
